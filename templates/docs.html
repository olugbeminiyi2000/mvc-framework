V1 MVC Framework

A personal MVC framework built from scratch — raw sockets, no external web framework, custom template engine, full validation and transaction layer.

Raw TCP sockets No external web framework JSON persistence ACID transactions Hot reload

The framework handles the infrastructure: socket server, HTTP parsing, routing, and response building. You define the Model, View, and Controller for each project you build on top of it. Every project is a plain Python package inside projects/ with five files.

Philosophy: The framework provides the structure. Your project provides the logic. Two people using this framework can build completely different things — a task manager, a blog, a user system — and neither project knows anything about the other.

Steps 4–6 are the MVC layer — the parts you implement per project.

Step by step

1. HTTP Server — receives the raw TCP connection, reads the full request including headers and body based on Content-Length.

2. Request Parser — extracts method, path, and body. Auto-detects Content-Type and normalises the body into a plain Python dict regardless of format.

3. Router — matches path + method to a registered route, instantiates the controller, calls the action, and passes the body dict as **kwargs.

4. Controller — runs your business logic, reads or writes the model, and returns a response dict.

5. View — receives the controller response as controller_response= in kwargs, renders it as HTML or JSON, and returns a string.

6. Response — the HTTP server wraps the string in a proper HTTP response with status, content-type, and charset headers, and sends it back to the client.

Quickstart

Every project lives in projects/your_project/ with these five files:

projects/
└── your_project/
    ├── __init__.py
    ├── model.py       # your data layer
    ├── controller.py  # your business logic
    ├── view.py        # your rendering
    └── router.py      # URL → action mappings

1 — Register your routes

Run once to save routes into router_state.pkl:

python -m projects.your_project.router

2 — Start the server

python -m servers.v1_runserver

Server starts at http://127.0.0.1:8080. Hot reload is on — the server restarts automatically whenever any .py file changes, with a 2-second debounce.

Controller

Inherit from V1AbstractController. Every action method receives the parsed request body as **kwargs and returns a value that the view will receive as controller_response. 

from controllers.v1_Controller import V1AbstractController

class MyController(V1AbstractController):
    def __init__(self):
        pass

    def my_action(self, **kwargs):
        name = kwargs.get("name", "World")
        return {"message": f"Hello, {name}!"}
Rule: Every action method must accept only self and **kwargs. The router validates this at add_route() time and raises a ValueError if you add required positional arguments.

Model

Wraps V1Model, which persists data to a JSON file. Pass a unique file_path per project to keep data separate across projects. 

from models.v1_Model import V1Model

db = V1Model(file_path="my_project.json")

# Create
db.add_key_value("items", [])

# Read
items = db.get_key_value("items")

# Update — pass the key as a keyword argument
db.update_key_value(items=updated_list)

# Delete a key
db.delete_key_value("items")
overwrite_data: This method replaces all stored data with whatever kwargs you pass, but it validates key uniqueness first — every key you provide must not already exist in the current data, or it raises a ValueError. To reset an existing key's value use update_key_value() instead.

Atomic writes

Every write goes to a temp file first, then atomically replaces the real file via os.replace(). Data is never left in a half-written state.

Transactions

For operations that must all succeed or all fail together. Pass your V1Model instance into Transactions — it wraps the model and intercepts every write:

from models.transaction.v1_Transaction import Transactions

db = get_model()
tx = Transactions(db)           # wrap your model instance

tx.begin_transaction()
tx.add("session_id", "abc123")  # add a new key (must not already exist)
tx.update("status", "active")   # update an existing key
tx.delete("temp")              # queue a deletion
tx.read("items")               # read a key; result via Transactions.get_read_data()
tx.preview_transactions()        # inspect the full queue before committing
tx.commit_transaction()          # apply all — auto-rolls back if any op failed
tx.end_transaction()             # close the transaction
All-or-nothing: if any queued operation fails validation, commit_transaction() automatically calls rollback_transaction() before returning False. You can also call rollback_transaction() yourself at any point before committing. Note: add requires the key to be absent from the model — if it already exists, add raises immediately and the whole transaction rolls back on commit. If the key is new, you can queue both add then update on it safely; update simply overwrites what add set and the last value wins.

Batch operations

Queue multiple operations in one call using process_batch():

tx.begin_transaction()
tx.process_batch([
    {"action": "add", "key": "name", "value": "Alice"},
    {"action": "update", "key": "score", "value": 42},
    {"action": "delete", "key": "temp"},
])
tx.commit_transaction()
tx.end_transaction()

Validation

Validation runs automatically on every model write — you do not call it explicitly. Every call to add_key_value(), update_key_value(), overwrite_data(), and transaction ops passes through two layers before anything is written to disk.

Built-in key rules

Every key is checked against these rules on every write. Any failure raises a ValueError before the data is touched:

Non-empty Key cannot be None, an empty string, or a falsy value.
Valid identifier Must pass str.isidentifier(). Use underscores instead of spaces.
Not a keyword Cannot be a reserved Python word (if, for, class, etc.).
Unique (on add) Key must not already exist when using add_key_value(). Use update_key_value() to change an existing key.
Length ≤ 300 Key must not exceed 300 characters.

Automatic format detection

Before writing, the framework inspects the value and tries to identify its format. If it looks like an email, phone number, or URL, the format is validated automatically — no extra code needed on your part:

Email Matched against RFC 5321-style regex. Invalid emails are rejected.
Phone Checked against international dialing code patterns. Must include country code.
URL Must start with http://, https://, ftp://, ws://, or wss://.
Non-string values (lists, dicts, ints, booleans) skip format detection entirely and pass through as-is, as long as they are not None.

Custom rules

Register your own per-field validation rules on the model. The rule is a plain Python function that takes the value and returns a boolean: 

db = get_model()

# Register a rule: priority must be one of three values
db.register_rule(
    "priority",
    lambda v: v in ("low", "medium", "high"),
    "priority must be low, medium, or high",
    "low"   # example value used to type-check the rule itself
)

# This will now raise ValueError for any write where priority is invalid
db.add_key_value("priority", "urgent")   # → ValueError
db.add_key_value("priority", "high")     # → OK

# Remove the rule later
db.delete_rule("priority")
Rules are persisted. Custom rules are saved to model_state.pkl and reloaded every time the model is instantiated. Register rules once — usually at the top of your model.py — and they stay active across server restarts.

View

Inherit from V1BaseView, set content_type, and implement render(). The controller response arrives as kwargs["controller_response"].

JSON view

from views.v1_View import V1BaseView

class MyJsonView(V1BaseView):
    content_type = V1BaseView.CONTENT_TYPES["JSON"]

    def __init__(self): pass

    def render(self, **kwargs):
        return self.render_json(kwargs.get("controller_response") or {})

HTML view

class MyHtmlView(V1BaseView):
    content_type = V1BaseView.CONTENT_TYPES["HTML"]

    def __init__(self): pass

    def render(self, **kwargs):
        data = kwargs.get("controller_response") or {}
        return self.render_template("my_template.html", data)
Templates live in the templates/ directory. Use {{key}} placeholders — the view replaces them with values from the dict you pass to render_template(). Unmatched placeholders are removed from the output automatically.

Nested templates

Templates can be organised into subdirectories. Pass the relative path from templates/ as the template name: 

# renders templates/emails/welcome.html
self.render_template("emails/welcome.html", data)

# renders templates/admin/dashboard.html
self.render_template("admin/dashboard.html", data)
Note: The framework blocks .. in template names to prevent path traversal. Subdirectory paths are safe as long as they do not contain ...

Available content types

V1BaseView.CONTENT_TYPES["HTML"]   # text/html
V1BaseView.CONTENT_TYPES["JSON"]   # application/json
V1BaseView.CONTENT_TYPES["PLAIN"]  # text/plain
V1BaseView.CONTENT_TYPES["XML"]    # application/xml
V1BaseView.CONTENT_TYPES["PDF"]    # application/pdf

Templates

Templates are plain HTML files that live in the templates/ directory at the root of your project. The framework has a built-in template engine that replaces {{key}} placeholders with values you pass in, then strips any unmatched placeholders from the output automatically.

Placeholder syntax

Inside any .html file in templates/, use double curly braces:

<!-- templates/profile.html -->
<h1>Hello, {{ name }}</h1>
<p>You have {{ count }} tasks pending.</p>
<p>Role: {{ role }}</p>  <!-- removed if not passed -->

Render it from your view by passing a dict whose keys match the placeholder names:

def render(self, **kwargs):
    data = kwargs.get("controller_response") or {}
    return self.render_template("profile.html", {
        "name":  data.get("name"),
        "count": data.get("count"),
    })
    # {{ role }} was not passed → removed from output automatically

Subdirectories

Templates can be organised into subdirectories inside templates/. Pass the path relative to templates/ as the template name: 

templates/
├── profile.html
├── emails/
│   ├── welcome.html
│   └── reset.html
└── admin/
    └── dashboard.html
self.render_template("emails/welcome.html", data)
self.render_template("admin/dashboard.html", data)
Security: The engine blocks any template name containing .. or starting with /. All other paths — including deeply nested ones — resolve safely inside templates/.

How it works internally

The engine reads the file as UTF-8, then runs a regex substitution for every key in the dict you pass. After all replacements, a second regex removes any remaining {{ unmatched }} placeholders so they never appear as raw text in the browser. No loops, conditionals, or inheritance — just straightforward string replacement.

Static Files

Any file placed inside the static/ directory at the project root is served directly by the HTTP server at the URL path /static/<filename>. No extra configuration needed.

Referencing static files in templates

<!-- stylesheet -->
<link rel="stylesheet" href="/static/style.css" />

<!-- script -->
<script src="/static/app.js"></script>

<!-- image -->
<img src="/static/images/logo.png" />

Subdirectories

Nested folders inside static/ are fully supported. The URL path mirrors the folder structure exactly: 

static/
├── style.css             # /static/style.css
├── app.js                # /static/app.js
├── images/
│   ├── logo.png          # /static/images/logo.png
│   └── banner.jpg        # /static/images/banner.jpg
└── fonts/
    └── mono.woff2        # /static/fonts/mono.woff2

Supported MIME types

.html text/html .css → text/css
.js application/javascript .svg → image/svg+xml
.png / .jpg image/png — image/jpeg .gif → image/gif
.ico image/x-icon .mp4 → video/mp4
Anything else falls back to application/octet-stream, so the browser will treat it as a binary download. Add your own MIME type mapping in servers/v1_HTTPServer.py inside get_content_type() if you need a type that isn’t listed above.

Router

Maps a URL path + HTTP method to a controller action and a view. Routes are saved to router_state.pkl and reloaded every time the server starts. 

from routers.v1_Router import V1Router

route = V1Router()
route.add_route(
    "/path",           # URL path
    MyController,      # controller class (not an instance)
    "action_name",    # method name on the controller
    MyView,            # view class (not an instance)
    "GET"             # HTTP method: GET POST PUT PATCH DELETE
)
One method per path. The router stores one entry per URL path. If you need GET and POST on the same resource, use distinct paths: /tasks for GET and /tasks/create for POST.

Server

Built on raw Python sockets — no Flask, no Django, no external web framework. The server binds to 127.0.0.1:8080 and handles one connection at a time. 

python -m servers.v1_runserver

Hot reload

Powered by Watchdog. Any .py file change triggers an automatic server restart with a 2-second debounce to prevent rapid restarts on multiple simultaneous saves.

Static files

Any file under static/ is served directly at /static/<path>. Nested subdirectories are fully supported — the URL path mirrors the folder structure. MIME types are auto-detected for HTML, CSS, JS, images, and video. 

# File on disk                   →  URL
static/docs.css                 # /static/docs.css
static/images/logo.png          # /static/images/logo.png
static/fonts/mono.woff2         # /static/fonts/mono.woff2
This page loads its stylesheet from /static/docs.css — the same static file server that you use for your own projects.

Content Types

The request parser reads the Content-Type header and normalises the body into a plain Python dict before it reaches the controller. Your controller always gets the same **kwargs regardless of how the client sent the data.

application/json Body parsed with json.loads(). Keys become kwargs directly.
multipart/form-data HTML form with file uploads. Text fields are strings; files are a dict: {"filename": "...", "content_type": "...", "data": b"..."}.
x-www-form-urlencoded Standard HTML form. Parsed with parse_qs(). Keys become kwargs.

Full Example — Tasks API

A complete project demonstrating all five HTTP methods. This project is live on this server — the same framework serving these docs is running the Tasks API below.

GET /tasks List all tasks
POST /tasks/create Create a new task
PUT /tasks/update Full replace — all fields required
PATCH /tasks/patch Partial update — send only what changes
DELETE /tasks/delete Remove a task by id

model.py

from models.v1_Model import V1Model

def get_model():
    return V1Model(file_path="tasks_model.json")

controller.py

from controllers.v1_Controller import V1AbstractController
from projects.tasks.model import get_model

class TaskController(V1AbstractController):
    def __init__(self): pass

    def list_tasks(self, **kwargs):
        tasks = get_model().get_key_value("tasks") or []
        return {"tasks": tasks, "count": len(tasks)}

    def create_task(self, **kwargs):
        title = kwargs.get("title")
        if not title:
            return {"error": "title is required"}
        db = get_model()
        tasks = db.get_key_value("tasks") or []
        new_id = max((t["id"] for t in tasks), default=0) + 1
        task = {"id": new_id, "title": title,
                "description": kwargs.get("description", ""),
                "status": kwargs.get("status", "pending"),
                "priority": kwargs.get("priority", "low")}
        tasks.append(task)
        if new_id == 1: db.add_key_value("tasks", tasks)
        else: db.update_key_value(tasks=tasks)
        return {"message": f"Task '{title}' created", "task": task}

    def update_task(self, **kwargs): ...   # PUT: all fields required
    def patch_task(self, **kwargs):  ...   # PATCH: any subset of fields
    def delete_task(self, **kwargs): ...   # DELETE: requires id

view.py

from views.v1_View import V1BaseView

class TaskJsonView(V1BaseView):
    content_type = V1BaseView.CONTENT_TYPES["JSON"]
    def __init__(self): pass
    def render(self, **kwargs):
        return self.render_json(kwargs.get("controller_response") or {})

router.py

from projects.tasks.controller import TaskController
from projects.tasks.view import TaskJsonView
from routers.v1_Router import V1Router

route = V1Router()
route.add_route("/tasks",        TaskController, "list_tasks",  TaskJsonView, "GET")
route.add_route("/tasks/create", TaskController, "create_task", TaskJsonView, "POST")
route.add_route("/tasks/update", TaskController, "update_task", TaskJsonView, "PUT")
route.add_route("/tasks/patch",  TaskController, "patch_task",  TaskJsonView, "PATCH")
route.add_route("/tasks/delete", TaskController, "delete_task", TaskJsonView, "DELETE")

curl Reference

All routes accept application/json, multipart/form-data, and application/x-www-form-urlencoded. Examples below use JSON.

GET List all tasks

curl http://localhost:8080/tasks

POST Create a task

curl -X POST http://localhost:8080/tasks/create \
  -H "Content-Type: application/json" \
  -d '{"title": "My task", "description": "Details", "status": "pending", "priority": "high"}'

PUT Full replace (all fields required)

curl -X PUT http://localhost:8080/tasks/update \
  -H "Content-Type: application/json" \
  -d '{"id": 1, "title": "Updated", "description": "New desc", "status": "done", "priority": "low"}'

PATCH Partial update

curl -X PATCH http://localhost:8080/tasks/patch \
  -H "Content-Type: application/json" \
  -d '{"id": 1, "status": "in-progress"}'

DELETE Remove by id

curl -X DELETE http://localhost:8080/tasks/delete \
  -H "Content-Type: application/json" \
  -d '{"id": 1}'
